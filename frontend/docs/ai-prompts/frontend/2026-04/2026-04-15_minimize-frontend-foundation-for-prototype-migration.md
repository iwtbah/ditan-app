# 最小化 frontend 底座以承接原型迁移

- 日期：2026-04-15
- 来源范围：用户在 Codex 会话中提供的第一阶段执行要求
- 主题标签：`frontend` `foundation` `cleanup` `migration` `routing`

## 原始提示词

基于以下正式工程目录进行改造：

/Users/zhaowenzhuo/Workspace/ditan-app/frontend

现在执行第一步：先最小化这个已有前端工程，让它变成一个适合承接原型代码迁移的干净底座。

目标：
- 保留现有技术栈和工程规范
- 删除或下线无关的示例代码、占位页面、测试页面、冗余样式、无用组件、无关状态
- 保留最基础的运行能力：应用入口、路由、全局样式、别名配置、必要的 providers、基础 layout 壳
- 输出一个“最小可运行”的 frontend 工程
- 暂时不要迁移原型代码
- 暂时不要做页面视觉设计

需要你做的事情：
1. 先完整分析 frontend 目录结构、依赖、路由结构、状态管理、样式体系、公共组件目录
2. 识别哪些是底座必须保留的，哪些是可删减的
3. 进行最小化改造，保留：
   - src/main 或入口文件
   - 路由基础结构
   - app/providers
   - 全局样式与主题基础
   - 基础路径别名
   - 基础 layout 外壳
4. 删除或隔离：
   - demo 页面
   - 与迪探当前原型无关的业务页面
   - 无引用组件
   - 无意义 mock 逻辑
   - 冗余工具函数
5. 确保项目可以正常启动
6. 输出：
   - 当前保留的工程骨架说明
   - 删除/保留清单
   - 后续最适合承接原型迁移的目录建议

约束：
- 不要引入新技术栈
- 不要提前大规模重命名目录
- 不要做和原型无关的抽象
- 所有改动以“干净、稳定、可迁移”为目标

最终请直接修改代码，并给出：
1. 改动摘要
2. 删除了哪些模块
3. 保留了哪些底座能力
4. 目前工程目录树（精简版）

## 补充上下文

- 当前阶段只做正式工程瘦身，不迁移原型代码。
- 路由路径应尽量保留未来承接原型的基础形态，但去掉当前演示业务实现。
- 所有改动都应围绕“干净、稳定、可迁移”的最小可运行底座。
