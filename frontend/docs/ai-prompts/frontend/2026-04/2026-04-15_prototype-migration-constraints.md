# 原型代码迁移到底座工程的约束与执行要求

- 日期：2026-04-15
- 来源范围：用户在 Codex 会话中提供的迁移目标与硬性要求
- 主题标签：`frontend` `migration` `architecture` `engineering` `prototype`

## 原始提示词

你现在是资深前端架构师 + React 工程化改造专家。

我有两个本地目录：

1. 原型设计代码目录：
/Users/zhaowenzhuo/Workspace/ditan-prototype
2. 迪探前端工程目录：
/Users/zhaowenzhuo/Workspace/ditan-app/frontend

目标不是重做页面，而是把“原型设计代码”无损迁移到“已有前端底座工程”中，并按照底座规范完成工程化改造。

硬性要求：
- 最终以 /Users/zhaowenzhuo/Workspace/ditan-app/frontend 作为唯一保留的正式工程
- 页面UI、布局、交互、文案、视觉层次、动效表现不能发生变化
- 允许做的事情：目录整理、路由接入、状态管理接入、样式归位、组件抽取、类型补全、公共方法抽离、资源整理、死代码清理
- 不允许做的事情：擅自改设计、擅自改字段语义、擅自重命名用户可见文案、擅自换视觉方案、擅自删页面
- 每一步修改前先分析目录结构和依赖
- 每一步修改后都要输出“变更清单、风险点、下一步建议”
- 每一步都优先做最小改动，不要一次性大重构
- 如果发现原型代码和底座工程技术实现冲突，优先保持页面效果一致，其次再做工程规范适配
- 所有新增和修改的代码都要可维护，命名清晰，目录合理
- 对于可复用但当前还不稳定的模块，不要过度抽象
- 不要只给建议，直接落地修改代码
请先记住我的目标与要求，接下来我分阶段告诉你怎么执行操作

## 补充上下文

- 本轮先记录总目标与边界，不提前进行大规模迁移。
- 后续每个阶段都要先对比原型目录与正式工程目录，再做最小改动落地。
- 若出现技术实现冲突，处理优先级为：页面效果一致 > 工程规范适配。
