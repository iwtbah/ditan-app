# 将 ditan-prototype 原型代码迁移接入 frontend

- 日期：2026-04-15
- 来源范围：用户在 Codex 会话中提供的第二阶段迁移要求
- 主题标签：`frontend` `prototype` `migration` `routing` `styles`

## 原始提示词

现在执行第二步。

源目录（原型代码）：
/Users/zhaowenzhuo/Workspace/ditan-prototype

目标目录（正式工程）：
/Users/zhaowenzhuo/Workspace/ditan-app/frontend

任务目标：
把 ditan-prototype 中的原型设计代码，尽可能无损地迁移到 frontend 工程中，并接入现有底座，让页面先在 frontend 中跑起来。

第一优先级：
- 页面效果一致
- 路由可访问
- 样式，交互不能修改

第二优先级：
- 代码目录尽量合理
- 尽量适配现有底座约定

注意：
这一步的重点是“迁移接入”，不是“深度优化重构”。
不要在这一步过度拆组件、过度重写样式、过度抽象逻辑。原型工程与底座工程的依赖版本和组件体系有差异，则以原型设计的版本为准。并删除底座的旧组件，避免产生无用垃圾

请按以下要求执行：
1. 分析 ditan-prototype 的页面结构、组件依赖、资源文件、样式方案、路由组织方式
2. 将原型中的页面、组件、静态资源、样式文件迁移到 frontend
3. 接入 frontend 已保留的路由系统
4. 对必须修改的导入路径、别名、静态资源路径、样式引用进行适配
5. 如果原型代码和底座存在冲突，优先保证页面效果不变，再做最低限度兼容
6. 对暂时不规范但影响运行的代码，先兼容，不要过度重写
7. 确保迁移后的页面在 frontend 里可以正常访问和渲染

迁移原则：
- 原型页面的布局、间距、颜色、字号、层级、文案、图片位置、卡片结构、交互入口都不能随意变化
- 原型中的页面命名、组件命名如果明显不合理，可以内部微调，但用户可见内容不能变
- 允许将页面放入约定目录，例如 pages / features / components，但要以最小迁移成本为前提
- 不要在这一步做大规模组件抽取
- 不要在这一步做视觉优化

迁移完成后，请输出：
1. 迁移了哪些页面
2. 迁移了哪些组件
3. 迁移了哪些资源
4. 做了哪些兼容适配
5. 当前还有哪些不规范但暂时保留的问题
6. 第三步适合优化的重点列表

最终要求：
- 直接落地代码
- 保证 frontend 可以运行
- 页面效果与 ditan-prototype 保持一致

## 补充上下文

- 迁移阶段优先保持原型的路由结构、视觉和交互表现。
- 与底座不兼容的地方允许先做兼容层，不在本轮进行深度工程化重写。
- 仅引入原型运行所需的页面、共享组件、样式与依赖，避免把未使用的原型垃圾一起带入正式工程。
